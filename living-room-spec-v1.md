# 리빙룸 (가칭) — 제품 스펙 & 작업 핸드오프 문서

> **한 줄 정의:** 내 컴퓨터를 "집"으로 취급해서, 내가 만든 이미지·문장·스크린샷·사이버 굿즈를 방마다 자유롭게 배치하고 꾸미는, 켤 필요 없이 늘 거기 있는 로컬 데스크탑 앱.
>
> 이 문서는 **처음 보는 사람도, 그리고 Claude Code도** 이걸 그대로 읽고 작업을 시작할 수 있게 쓰였다. 위에서부터 순서대로 읽으면 된다.

---

## 0. 이 문서 읽는 법 (개발 처음인 사람용)

- **1~3장**은 "무엇을, 왜 만드는가"다. 코드 안 나온다. 먼저 읽어서 그림을 잡는다.
- **4~6장**은 "어떻게 만드는가"다. 기술 스택, 데이터 구조, 폴더 구조.
- **7장**이 핵심이다. **작업을 7개의 작은 관문(게이트)으로 쪼갰다.** 한 게이트 = 그 자체로 완성되고 눈에 보이는 결과물. 순서대로 하나씩 통과하면 된다.
- **8장 이후**는 v1 이후의 미래 계획이라 지금 안 봐도 된다.
- 모르는 단어가 나오면 **부록 A 용어 사전**을 본다.

## 1. 린 캔버스 (Lean Canvas)

| 블록 | 내용 |
|---|---|
| **문제 (Problem)** | ① 그동안 만든 트래커/도구들이 "내가 찾아가서 켜고 입력해야 하는 별도의 장소"라 마찰이 커서 안 쓰게 됨. ② 아끼는 자료(RP 로그, 스크린샷, OC 이미지, 멋진 문장)가 트위터·드라이브·폴더 등으로 **파편화**되어 한눈에 못 봄. ③ 기존 꾸미기/전시 수단은 웹 플랫폼에 의존하거나 범용이라, "내 것"이라는 느낌이 약하고 아이디어가 쉽게 베껴짐. |
| **고객 (Customer Segments)** | 1차: 제작자 본인. 2차: 자기 OC·RP·세션 스크린샷을 아끼고 전시·수집하고 싶은 오타쿠/TRPG 창작자. |
| **고유 가치 (UVP)** | **"트래커가 아니라, 돌아가고 싶은 집."** 켤 필요 없이 늘 떠 있고, 의무가 아니라 애착으로 여는, 내 이미지로 직접 짓는 디지털 방. |
| **해결책 (Solution)** | 상시 떠 있는(ambient) 로컬 데스크탑 앱. "방" 단위 디오라마. 내 이미지 드래그 배치 + 가벼운 인앱 편집(크롭·색조). 문장을 액자처럼 꾸미기. 내 이미지로 사이버 굿즈(포토카드·아크릴 스탠드 등) 찍어내 진열. |
| **채널 (Channels)** | 트위터 개발 공계, Postype, GitHub Releases(설치 파일 배포 + 자동 업데이트). |
| **수익 (Revenue)** | 없음(무료·비영리). 원하면 후원 링크 정도만. |
| **비용 (Cost)** | 사실상 0. 완전 로컬, 외부 API 없음, 서버 없음, GitHub 무료. |
| **핵심 지표 (Key Metrics)** | "내가 일주일에 며칠 *자발적으로* 여는가"(애착 지표). 보조: 올린 오브젝트 수, 만든 방 수, (배포 시) 설치·재방문 수. |
| **불공정 우위 (Unfair Advantage)** | 제작자의 일러스트/디자인 제작력 + TRPG 깊이 + **"내 OC를 올리는 캔버스"라는 구조** → 컨셉은 베껴도 그 안의 콘텐츠는 각자 자기 것이라 원본성이 보존됨. |

## 2. 설계 철학 (이걸 어기면 망한다)

이 프로젝트가 과거의 트래커들처럼 죽지 않으려면, 모든 기능 결정은 아래 3원칙을 통과해야 한다.

1. **켤 필요가 없다 (Ambient).** "사용하려고 실행한다"는 마찰이 실패의 주원인이었다. 앱은 배경처럼 늘 거기 있고, 사용자는 *들르고 싶을 때* 들른다. → **트레이 상주 + 바탕화면 바로가기, 부팅 시 자동 실행(옵션).**
2. **의무가 아니라 보상이다 (Reward, not chore).** 무언가를 "기록·체크"하러 여는 게 아니라, **구경하고 자랑하러** 연다. 기능은 "해야 할 일"을 만들면 안 되고 "또 보고 싶은 것"을 만들어야 한다.
3. **내 이미지로 꾸미는 게 핵심 기능이다.** 사용자의 이미지 파일을 올리고 배치·편집·전시하는 경험이 앱의 심장이다. 다른 모든 기능은 이걸 거든다.

> **안티 패턴(하지 말 것):** 할 일 목록, 강제 알림, "오늘 입력 안 했어요" 같은 죄책감 유발, 통계 대시보드. 이건 전부 트래커의 저주로 되돌아가는 길이다.

## 3. 핵심 컨셉: "컴퓨터 = 집"

집은 여러 **방(Room)**으로 이루어진다. 방마다 성격이 다르지만, **속을 까보면 모든 방은 같은 엔진을 쓴다.**

> **엔진의 정체:** 방이란 "꾸밀 수 있는 오브젝트(이미지·문장 카드·굿즈)들이 자유롭게 놓인 하나의 캔버스(씬)"다. 엔진은 하나, 방은 그 엔진의 인스턴스가 여러 개일 뿐. 그래서 방을 추가해도 코드가 비대해지지 않고, 기능은 "새 오브젝트 타입"이나 "새 방"을 더하는 식으로 자란다.

방 목록(전체 로드맵):

| 방 | 역할 | 버전 |
|---|---|---|
| **거실 (Living Room)** | 내 이미지·스크린샷을 자유롭게 배치해 꾸미는 메인 공간 | **v1** |
| **진열장 / 서재 (Shelf / Study)** | 아낀 RP·문장을 "책/액자"로 꽂아 전시. 문장 꾸미기 | v2 |
| **공방 / 아틀리에 (Workshop)** | 내 이미지로 사이버 굿즈(포토카드·아크릴 스탠드 등) 제작 → 진열 | v3 |
| **현관 / 문패 (Entrance)** | 들어오면 맞아주는 곳. 날짜·계절감 정도의 은은한 분위기(트래커 아님) | v4+ |
| **수납장 / 던지는 상자 (Inbox)** | 아무거나 0마찰로 던져 넣는 칸. 정리는 내킬 때. 파편화 해소 | v4+ |
| **분위기 (Ambience)** | 방별 색감·조명, 선택적 잔잔한 BGM | 폴리시 |

## 4. 기술 스택

| 항목 | 선택 | 이유 |
|---|---|---|
| **앱 프레임워크** | **Tauri (v2, 현재 stable / 설치 시 버전 고정)** | Electron과 달리 크롬을 통째로 안 싣고 OS 내장 웹뷰를 씀 → 메모리 가벼움(아이들 50~100MB대). 결과물은 네이티브 데스크탑 앱(브라우저·서버 없음). 자동 업데이트 플러그인 제공. |
| **캔버스/렌더링** | **Konva.js (v9.x)** | 이미지 드래그/리사이즈/회전(Transformer), 색조·밝기·채도·대비 필터, 크롭을 거의 **내장**으로 제공. 우리가 짤 코드가 확 줄어 "미완성 방치" 위험이 낮음. |
| **UI 레이어** | 바닐라 JS 또는 가벼운 프레임워크(선택) | v1은 화면이 단순해서 바닐라 + Konva로 충분. 굳이 React 안 써도 됨. (쓰고 싶으면 가벼운 거 1개까지.) |
| **저장** | 로컬 파일시스템 (Tauri fs API) | DB 불필요. 방 상태 = JSON 파일, 이미지 = 앱 데이터 폴더 안의 실제 파일. |
| **언어** | 프론트는 JS, 백엔드 Rust는 거의 안 건드림 | 파일 저장·다이얼로그·업데이트 전부 Tauri의 JS API로 가능. Rust는 설정 파일 손보는 수준. |
| **배포** | GitHub Releases + Tauri updater | 설치 파일 올리면 사용자 컴퓨터에서 자동 업데이트. |

> **솔직한 대안 메모:** "웹기술에 1도 안 닿고 진짜 게임 같은 방"을 원하면 **Godot**도 가능하다. 다만 인앱 편집·자동 업데이트를 직접 만들어야 해서 손이 더 간다. 우리는 *편집 기능이 거의 공짜로 따라오는* Tauri+Konva로 간다. (Tauri 속은 웹기술이지만, 사용자가 싫어했던 "웹을 켜야 함"은 없으므로 진짜 니즈는 충족된다.)

### 4-1. 성능 3원칙 (램 15GB 노트북에서 안 버벅이게)

1. **올릴 때 자동 축소.** 큰 이미지는 가져오는 순간 **최대 변 1500px**로 줄여서 저장한다(원본은 굳이 보관 안 함).
2. **정적 이미지만.** APNG·동영상은 받지 않는다(합의됨). PNG/JPG/WebP만. → 메모리·CPU 안정.
3. **JSON엔 좌표만, 이미지는 파일로.** 이미지를 base64로 JSON에 박지 않는다. 이미지는 앱 폴더에 파일로 두고, 방 JSON에는 "어느 파일을, 어디에, 어떤 필터로" 같은 **값만** 저장한다. (+ 썸네일 캐싱하면 수십 장 올려도 가볍다.)

## 5. 데이터 모델

### 5-1. 앱 데이터 폴더 구조

Tauri의 `appDataDir`(OS별 표준 앱 데이터 경로) 아래에 둔다.

```
<appDataDir>/
├─ rooms/
│   └─ living-room.json        # 거실 방 상태 (좌표·필터값만)
├─ assets/
│   └─ images/
│       ├─ <uuid>.webp         # 사용자가 올린 이미지(자동 축소본)
│       └─ <uuid>.thumb.webp   # (선택) 썸네일
└─ settings.json               # 전역 설정(테마, 자동실행 등)
```

### 5-2. 방(Room) JSON 스키마

```jsonc
{
  "schemaVersion": 1,
  "id": "living-room",
  "name": "거실",
  "background": {
    "type": "theme",          // "theme" | "color" | "image"
    "value": "warm"           // 테마 이름 / "#f3e9dd" / "assets/images/<uuid>.webp"
  },
  "objects": [
    {
      "id": "f1a2...",         // uuid
      "type": "image",         // v1은 "image"만. v2:"text", v3:"goods" 추가
      "src": "assets/images/<uuid>.webp",
      "x": 120, "y": 80,       // 좌상단 좌표
      "width": 300, "height": 200,
      "rotation": 0,           // 도(degree)
      "scaleX": 1, "scaleY": 1,
      "zIndex": 3,             // 앞뒤 순서 (클수록 앞)
      "crop": null,            // null 또는 {x,y,width,height}
      "filters": {             // 기본값 0 = 원본
        "brightness": 0,       // -1.0 ~ 1.0
        "saturation": 0,       // -1.0 ~ ...
        "hue": 0,              // 0 ~ 360
        "contrast": 0
      },
      "locked": false          // true면 드래그/편집 잠금
    }
  ]
}
```

> **스키마 버전(`schemaVersion`)을 반드시 둔다.** 나중에 구조가 바뀌어도 옛 저장 파일을 마이그레이션할 수 있게 하기 위함. v1에선 그냥 `1`로 박아두고 읽을 때 확인만 한다.

### 5-3. 좌표·편집은 "비파괴(non-destructive)"로

크롭·색조 등은 **원본 이미지 파일을 바꾸지 않는다.** JSON의 `crop`/`filters` 값으로만 표현하고, 화면에 그릴 때 Konva가 그 값을 적용해 렌더한다. → 언제든 되돌리기 가능, 파일 안 망가짐.

## 6. 화면 구조 (v1 거실)

```
┌─────────────────────────────────────────────┐
│ [상단 바] 방이름  | 배경테마 ▾ | 저장됨 ●     │
├───────┬─────────────────────────────────────┤
│ 좌측  │                                       │
│ 패널  │        캔버스 (Konva Stage)           │
│ ───── │   - 배경 레이어                       │
│ +이미 │   - 오브젝트 레이어(이미지들)         │
│  지   │   - 트랜스포머 레이어(선택 핸들)      │
│ 잠금  │                                       │
│ 삭제  │   (이미지를 여기로 드래그&드롭 가능)  │
│ 앞/뒤 │                                       │
│ 편집  │                                       │
└───────┴─────────────────────────────────────┘
```

- **선택**하면 트랜스포머(크기·회전 핸들)가 뜬다.
- **편집** 버튼 → 크롭/색조 패널이 열린다.
- 저장은 **자동**(변경 후 일정 시간 뒤 자동 저장, "저장됨 ●" 표시).

## 7. 작업 테크트리 — v1 거실 (핵심)

각 단계 = **그 자체로 완성되고 눈에 보이는 결과물**. 순서대로 하나씩 통과한다. 각 단계의 **[게이트]**가 "이 단계 끝났다"의 판정 기준이다. Claude Code에 넘길 땐 이 단계들을 위에서부터 하나씩 주면 된다.

### 단계 1 — 환경 세팅 & 빈 창
**할 일**
- Node.js, Rust 설치 (Tauri 요구사항).
- Tauri v2 프로젝트 생성(공식 create 명령), 빈 창 띄우기.
- Git 저장소 초기화 + GitHub 비공개 레포 생성, 첫 커밋.
- 빌드해서 바탕화면 바로가기로 빈 창이 뜨는지 확인.

**[게이트]** 바탕화면 바로가기를 더블클릭하면 빈 앱 창이 뜬다. GitHub에 코드가 올라가 있다.

---

### 단계 2 — 이미지 한 장 다루기 (Konva 기본)
**할 일**
- Konva 설치, Stage + Layer 구성.
- 테스트 이미지 1장을 코드로 띄운다.
- `draggable: true`로 드래그.
- Konva `Transformer`를 붙여 크기·회전.

**[게이트]** 창 안의 이미지 하나를 마우스로 끌고, 모서리로 키우고, 돌릴 수 있다.

---

### 단계 3 — 내 이미지 여러 장 올리기 (드래그&드롭 + 자동 축소)
**할 일**
- 바탕화면/폴더에서 PNG·JPG를 캔버스로 **드래그&드롭**해서 올린다. (+ "이미지 추가" 버튼으로 파일 선택도)
- 올리는 순간 **최대 변 1500px로 자동 축소**(offscreen canvas로 리사이즈) 후 `assets/images/`에 `.webp`로 저장.
- 오브젝트 여러 개를 각각 선택·삭제·앞뒤 순서(zIndex) 변경.

**[게이트]** 내 PNG 여러 장을 끌어다 놓고, 각각 자유롭게 배치·선택·삭제·앞뒤 정렬할 수 있다. 큰 사진을 올려도 자동으로 작아진다.

---

### 단계 4 — 저장 / 불러오기
**할 일**
- 방 상태를 `rooms/living-room.json`으로 저장(좌표·필터값만, 5-2 스키마).
- 앱 시작 시 JSON을 읽어 방을 복원(이미지는 `src` 경로에서 로드).
- 변경 후 자동 저장(디바운스) + 상단에 "저장됨 ●".

**[게이트]** 배치하고 앱을 껐다 켜도 방이 그대로 복원된다.

---

### 단계 5 — 가벼운 인앱 편집 (크롭 + 색조)
**할 일**
- 선택한 이미지에 **편집 패널**: 밝기·채도·색조(hue)·대비 슬라이더 → Konva 필터 적용(`node.cache()` 후 filters).
- **크롭**: 간단한 사각형 크롭 UI → Konva `crop` 속성에 반영.
- 전부 **비파괴**(JSON 값만 바뀜, 원본 파일 불변), "원래대로" 버튼.

**[게이트]** 올린 사진의 색조를 바꾸고, 일부를 잘라낼 수 있고, 되돌릴 수 있다.

---

### 단계 6 — 방 배경 & 테마
**할 일**
- 배경 레이어 추가: **SVG 또는 CSS 그라데이션 + 선반 한 줄** 정도(가볍게).
- 테마 1~2개(예: warm/night) — 벽 색·바닥만 바뀌는 수준.
- 상단 바에서 테마 전환.

**[게이트]** 캔버스가 빈 흰판이 아니라 "방"처럼 보인다. 테마를 바꿀 수 있다.

> **주의:** 여기서 배경 일러스트에 공들이지 말 것. 구조적 배경은 가벼운 SVG/CSS로 충분. 직접 그린 방 그림이나 Claude Design 레퍼런스는 나중 "폴리시" 단계에서 히어로 룸으로 얹는다.

---

### 단계 7 — 배포 + 자동 업데이트
**할 일**
- Tauri 빌드로 설치 파일 생성(Windows 우선).
- GitHub Releases에 업로드.
- Tauri **updater 플러그인** 설정(서명 키 생성, 업데이트 엔드포인트 = GitHub Releases).
- 트레이 상주 + 바탕화면 바로가기 확정.

**[게이트]** 친구가 설치 파일로 깔아서 켤 수 있고, 내가 새 버전을 올리면 그 사람 컴퓨터에서 자동으로 업데이트된다.

## 8. v2 이후 로드맵 (지금은 안 봐도 됨)

방을 **추가**하는 방식으로 자란다. 엔진(오브젝트+캔버스)은 그대로 두고 "새 오브젝트 타입"과 "새 방"만 더한다.

### v2 — 진열장 / 서재 (오브젝트 타입 `text` 추가)
- **문장 카드:** 아낀 RP·문장을 카드로 만들어 배치. **문장 스타일러** — 폰트(Pretendard/Noto 등), 색, 정렬, 배경지(한지/카드/네온), 장식 프레임. "문장을 굿즈처럼."
- **책장:** 저장한 글이 책등(spine)으로 선반에 꽂히고, 클릭하면 펼쳐 읽기. (기록함이 아니라 *구경하는 서재*.)

### v3 — 공방 / 아틀리에 (오브젝트 타입 `goods` 추가) — 리워드 루프의 심장
- 내 이미지로 **사이버 굿즈** 제작: 포토카드, 아크릴 스탠드(선반에 세워짐), 캔뱃지, 스티커 시트, 키링.
- 굿즈 틀(템플릿) + 내 이미지 → 합성 → 진열장/거실에 진열. "내 OC 포카를 찍어 모은다"가 또 열고 싶은 동기를 만든다.

### v4+ — 집을 완성하는 방들
- **현관/문패:** 들어오면 날짜·계절감만 은은히(트래커 아님).
- **수납장/던지는 상자:** 아무거나 0마찰로 드래그 투척 → 정리는 내킬 때. 파편화 해소의 핵심 칸.

### 폴리시 — 분위기
- 방별 색감·조명, 선택적 잔잔한 BGM(끌 수 있게). "와" 담당이지만 우선순위 최하.

## 부록 A. 용어 사전 (처음 보는 사람용)

- **Tauri:** 웹기술로 만든 화면을 가볍고 작은 네이티브 데스크탑 앱으로 포장해주는 도구. (Electron보다 가볍다.)
- **Konva.js:** 캔버스 위에 이미지·도형을 올리고 드래그·회전·필터를 쉽게 다루게 해주는 자바스크립트 라이브러리.
- **Stage / Layer:** Konva에서 Stage는 전체 캔버스 무대, Layer는 그 안의 레이어(층). 보통 배경 층, 오브젝트 층, 핸들 층으로 나눈다.
- **Transformer:** 선택한 오브젝트에 붙는 크기·회전 핸들(네모 모서리 손잡이).
- **오브젝트(Object):** 방에 놓인 꾸밀 수 있는 한 덩어리(이미지 한 장, 문장 카드 하나, 굿즈 하나).
- **방(Room) / 씬(Scene):** 오브젝트들이 놓인 하나의 캔버스. 거실·진열장 등이 각각 하나의 방.
- **비파괴 편집:** 원본 파일을 안 바꾸고, "어떻게 보일지" 값만 따로 저장해서 그릴 때 적용하는 방식. 언제든 되돌릴 수 있다.
- **appDataDir:** 앱이 자기 데이터를 저장하는 OS 표준 폴더 경로.
- **zIndex:** 겹쳤을 때 앞뒤 순서. 클수록 앞에 보인다.
- **디바운스(debounce):** 변경이 멈춘 뒤 잠깐 기다렸다가 한 번만 실행(자동 저장에 사용).
- **updater:** 새 버전을 올리면 사용자 앱이 알아서 받아 갱신하게 해주는 자동 업데이트 장치.
- **게이트:** 이 문서에서 "이 단계 끝났다"를 판정하는 합격 기준.

## 부록 B. Claude Code에 넘기는 법

1. 이 문서 전체를 프로젝트에 넣고 "이 스펙대로 만들 거야. **7장 단계 1**부터 시작하자"라고 준다.
2. **한 번에 한 게이트만** 시킨다. 각 단계의 [게이트]를 통과하면 커밋하고 다음 단계로.
